# API and Integration Decisions

This document records the currently implemented API exposure and integration assumptions as observed in the codebase.

## Public Edge Endpoints

Public here means "reachable through the Nginx edge service on `http://localhost:8000`".

| Path | Methods | Auth required | Notes |
| --- | --- | --- | --- |
| `/auth/login` | `POST` | No | Login endpoint generated by `fastapi-users`. Protected by Nginx rate limiting. |
| `/auth/logout` | `POST` | Yes | Exposed through the `/auth/` prefix and requires bearer auth. |
| `/alerts` | `GET` | Yes | Primary read API for stored alerts. |
| `/sync` | `POST` | Yes | Manual sync trigger. |
| `/health` | `GET` | Yes | Health and ingestion diagnostics. |

## Private or Internal Endpoints

Private here means "implemented by the FastAPI app but not exposed by the Nginx edge config".

| Path | Methods | Auth required | Notes |
| --- | --- | --- | --- |
| `/users/me` | `GET`, `PATCH` | Yes | Internal user self-service route from `fastapi-users`. |
| `/users/{id}` | `GET`, `PATCH`, `DELETE` | Yes | Internal user-management route; treat as operational/admin surface. |
| `/metrics` | mounted app | No app-level auth | Available on the API service itself, not through Nginx. |
| `/ping` | `GET` | No | Internal liveness check used by Compose health checks. |
| `/docs` | `GET` | No | FastAPI Swagger UI on the API service only. |
| `/redoc` | `GET` | No | FastAPI ReDoc on the API service only. |
| `/openapi.json` | `GET` | No | FastAPI OpenAPI schema on the API service only. |

## Authentication Decision

- The service uses database-backed bearer-token authentication via `fastapi-users`.
- Login is handled through `POST /auth/login` with form fields `username` and `password`.
- Protected routes use `Authorization: Bearer <token>`.
- Access tokens are persisted in the database and currently do not expire automatically (`lifetime_seconds=None`).
- Business endpoints require an active user.
- The upstream external mock uses a separate token scheme: `Authorization: Token <token>`.

## Rate Limits

- The only explicit rate limit in the repository is at the Nginx edge for `/auth/*`.
- Configured limit: `5` requests per minute per client IP.
- Configured burst: `3` extra requests with `nodelay`.
- Exceeded requests return HTTP `429`.
- No explicit rate limits are configured for `/alerts`, `/sync`, or `/health`.

## External Mock Contract Assumptions

The collector currently integrates with `GET /alerts/` on the mock upstream service.

### Required header

- `Authorization: Token <EXTERNAL_SERVICE_TOKEN>`

### Supported query parameters

| Parameter | Used by collector | Notes |
| --- | --- | --- |
| `since` | Yes | ISO8601 datetime. |
| `up_to` | Yes | ISO8601 datetime. |
| `source` | No | Optional comma-separated source filter supported by the mock. |

### Validation and defaults

- Invalid `since` or `up_to` values return `422`.
- `since > up_to` returns `422`.
- Unknown `source` values return `422`.
- If only `since` is provided, the mock defaults `up_to` to "now".
- If only `up_to` is provided, the mock defaults `since` to `up_to - 30 days`.
- If neither is provided, the mock defaults the window to the last `30` days.

### Error ratio behavior

- The mock can return random HTTP `500` responses based on a configured error probability.
- In local Docker Compose, `MOCK_ERROR_PROBABILITY` from the repo `.env` is passed into the mock container as `ERROR_PROBABILITY`, defaults to an error rate of 0.2.

### Supported `source` values

- `crowdstrike_falcon_edr`
- `microsoft_defender_for_identity`
- `palo_alto_ngfw`
- `cloudflare_waf`
- `aws_guardduty`
- `okta_threatinsight`
- `proofpoint_email_protection`
- `splunk_enterprise_security`
- `tenable_vulnerability_management`
- `recorded_future_intelligence_cloud`

## Health Endpoint Decision

The `GET /health` endpoint is the operational status endpoint for the collector. It is intended to summarize whether the system can reach its database, whether sync activity is still happening recently enough, and whether ingestion quality and latency remain within acceptable limits.

### Health data sources

- Database health comes from a direct PostgreSQL probe using `SELECT 1`.
- Sync recency and ingestion error rate come from `worker_executions` records stored in the application database.
- External mock latency comes from Prometheus, using the histogram exported by the API service for `external_alerts_call_duration_seconds`.
- Retry attempts are deduped by `sync_run_id`, keeping only the latest attempt for each logical sync run before status is computed.

### Health metrics used

- `database.status`: `up` when the direct DB probe succeeds, otherwise `down`.
- `database.latency_ms`: latency of the direct DB probe.
- `last_successful_sync`: latest successful deduped sync execution observed in the recent window.
- `error_rate_last_hour`: failed deduped sync runs divided by all deduped sync runs in the last hour.
- `p95_external_latency_seconds_last_hour`: Prometheus p95 over successful external mock call latency for the last hour.
- `recent_errors`: up to 10 most recent failed deduped sync executions.

### Status definitions

#### `ok`
- Returned when all of the following are true:
- The database probe is `up`.
- There is at least one successful sync in the last 3 hours.
- The latest successful sync is not stale relative to `HEALTH_SUCCESS_STALE_MINUTES` (default 30 minutes).
- `error_rate_last_hour` is below `HEALTH_ERROR_RATE_WARN` (default 0.20).
- `p95_external_latency_seconds_last_hour` is below `HEALTH_P95_WARN_SECONDS` (default 2.0 seconds).
- Prometheus latency data is available.

#### `degraded`
- Returned when the system is still functioning but one or more warning conditions are present:
- Prometheus latency data is unavailable, even if other checks pass.
- The latest successful sync is older than `HEALTH_SUCCESS_STALE_MINUTES` but there is still a successful sync within the last 3 hours.
- `error_rate_last_hour` is at or above `HEALTH_ERROR_RATE_WARN` but below `HEALTH_ERROR_RATE_DOWN`.
- `p95_external_latency_seconds_last_hour` is at or above `HEALTH_P95_WARN_SECONDS` but below `HEALTH_P95_DOWN_SECONDS`.

#### `down`
- Returned when any hard-failure condition is present:
- The database probe is `down`.
- There is no successful sync in the last 3 hours.
- `error_rate_last_hour` is at or above `HEALTH_ERROR_RATE_DOWN` (default 0.50).
- `p95_external_latency_seconds_last_hour` is at or above `HEALTH_P95_DOWN_SECONDS` (default 5.0 seconds), when Prometheus latency data is available.
